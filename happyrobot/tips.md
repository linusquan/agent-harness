Take-Home Checklist for Success
How to actually pass (and avoid hidden failure traps)

🎯 The Real Goal
This is not about building something that:
works
looks decent
is technically correct
👉 It’s about building something that is:
Useful, integrated, and credible as a real product

🧠 The Winning Formula
A strong submission =
 Agent (usable) + Dashboard (actionable) + Clear business value + Tight integration
If any of these are missing → high risk of rejection

🏆 What Actually Matters (in order)
1) Dashboard = Business tool, not data display
This is the #1 filter.
Strong:
Metrics tied to outcomes (revenue, conversion, cost, efficiency)
Helps answer: “What should I do next?”
Shows impact of the agent (not just activity)
Feels like a tool for a decision-maker (Sales, Ops, etc.)
Weak (very common):
Minimal KPIs
Data-science style (charts without decisions)
No link to business impact
No connection to the agent
👉 A technically correct dashboard can still fail if it’s not business-oriented.

2) Agent = Design matters more than existence
A working agent is expected. What matters is how it behaves.
Strong:
Handles back-and-forth interaction
Has thoughtful prompting (clear instructions, context awareness)
Produces consistent, useful outputs
Includes relevant integrations (if needed)
Weak (subtle but critical):
Prompting is too minimal → poor performance
Rigid or overly scripted
Not connected to real data
Missing key integrations
👉 A “working” agent with weak prompting or no integrations = rejection risk

3) Integration = Where many good candidates fail
This is a hidden but critical dimension.
Strong:
Agent outputs feed into the dashboard
Dashboard reflects agent performance
Feels like one system
Weak:
Agent and dashboard are separate
No shared data or feedback loop
Feels like stitched-together components
👉 Lack of integration = product feels fake → often rejected

4) Product Thinking = Make it make sense
Strong:
Clear:
User
Problem
Why it matters
Features tied to real use cases
Weak:
No clear use case
Overly generic solution
No explanation of value

5) Communication = Clear, but not enough alone
You can have:
Great video
Clear explanations
…and still get rejected.
👉 Why?
 Because communication cannot compensate for weak product decisions

⚠️ Hidden Failure Modes (From Real Candidates)
Even strong-looking submissions failed because:
❌ “Looks good, but…”
Good presentation + weak product → ❌
Good dashboard + no integration → ❌
Good agent structure + weak prompting → ❌
Good data analysis + no business value → ❌

❌ Data science ≠ product
Complex charts
Interesting analysis
…but:
No decisions enabled
No business impact
👉 This is a frequent rejection pattern

❌ “Barely enough” everywhere
Agent works
Dashboard exists
Write-up is okay
…but everything is shallow
👉 Signals low ownership → rejection

✅ What “Pass” Actually Looks Like
Agent behaves realistically and produces useful outputs
Prompting is thoughtful enough to get good results
Dashboard shows clear business impact
Agent + dashboard are connected
Clear explanation of who it’s for and why it matters

🚀 How to Upgrade Your Submission (High ROI)
1) Fix your prompting (huge impact)
Add clear instructions
Define tone, goals, constraints
Include context awareness
👉 Weak prompting = weak product

2) Make everything connected
Ask:
Does my dashboard reflect what my agent is doing?
Can I trace: agent action → metric → business outcome?

3) Turn data into decisions
Bad:
“Here are 5 charts”
Better:
“This shows which segment converts best → prioritize them”

4) Add one real integration (if relevant)
CRM, email, data source, etc.
 👉 Makes the product feel real

5) Simplify and focus
Better to do one use case really well
 than many shallow ones

🧪 Final 5-Minute Self-Test
If you fail any of these, you’re at risk:
Can I clearly explain who this is for and why they care?
Does my dashboard help someone make a decision?
Does my agent produce consistently good outputs?
Are agent + dashboard actually connected?
Does this feel like a real product, not a demo?

💡 Bottom Line
Most candidates don’t fail because they lack technical skills.
They fail because:
The product is not useful enough
The system is not integrated
The thinking is too shallow
👉 The bar is:
Build something a customer could actually use—and trust.
